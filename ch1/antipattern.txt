리액트의 대표적인 안티패턴에 대해 알아보자

------------------------------------------------------------------------------------

1. props drilling
- props가 부모 컴포넌트에서 자식컴포넌트로 여러 중간 컴포넌트를 거쳐서 전달되는 현상
- 중간 컴포넌트는 안쓰는 props를 내려줘야함
- 코드 이해를 방해하고 디버깅을 어렵게 함

context API쓰던가 atom기반의 솔루션을 도입하면 되긴함
근데 컴포넌트들에 여기에 종속적이게 되는게 좀 아쉽지

------------------------------------------------------------------------------------

2. 컴포넌트 내부의 convert
- 컴포넌트 내부에 복잡한 데이터 변환 로직을 작성
- API나 BE에서 전달받은 데이터는 FE에 적합하지 않을 수 있음

이렇게하면...
- 컴포넌트가 너무 많은걸 해서 역할 분석이 어렵
- 재사용성이 떨어짐 (다른 컴포넌트에서도 비슷한 일을 해야할 수있는데 복붙??)
- 변환로직을 고려해야되서 테스트가 어렵다.

따라서
!! 변환로직을 컴포넌트와 분리하자 !!
!! 유틸리티 함수 / hooks     !!

------------------------------------------------------------------------------------

3. 뷰 레이어에서 복잡한 로직을 작성
- 컴포넌트는 프레젠테이션에 집중해야함
- 하지만 뷰 레이어에서 비즈니스 로직을 다루는 함정에 빠지기 쉽다
- 컴포넌트가 크고 복잡해져서 테스트와 재사용성을 해침

컴포넌트의 재사용성 / 테스팅 / 유지보수가 어려워짐

------------------------------------------------------------------------------------

4. 중복된 코드
- 유사/동일한 코드조각이 app 여러곳에 흩어져있는게 흔함
- 코드베이스를 부풀리고 잠재적인 문제를 유발

이렇게 되면... 버그 수정/개선시 중복된 부분을 모두 고쳐야 함 

따라서....
- 공통로직을 utils / hooks / hoc로 관리하여 중앙/단일화

Single Source Of Truth
- 정보나 로직을 하나의 출처에서만 공급
- 데이터 불일치를 개선할 수 있음

------------------------------------------------------------------------------------

5. 기능을 너무 많이 가진 컴포넌트
- 컴포넌트에 기능이 많아서 맡고있는 책임이 늘어나면 거대한 덩어리가 되어버림
- 유지보수 / 테스트 / 이해가 난해

여러 컴포넌트로 쪼개거나 hooks로 분리가 필요

------------------------------------------------------------------------------------

- 인터페이스를 통해 모듈간에 발생하는 상호작용을 중심으로 소프트웨어를 구성

헤드리스 컴포넌트
- 렌더링 역할을 하지 않고 상태/로직을 관리하는 컴포넌트
- 렌더링 역할은 사용하는 컴포넌트가 알아서 하셈
