안티패턴

- 기능이 늘어날수록 유지보수를 어렵게 만들거나 오류로 이어지기 쉬운 패턴

------------------------------------------------------------------------------------

서버 상태: BE서버 or API같은 외부소스로 부터 가져온 데이터
로컬상태 : APP내에서 관리하는 데이터

서버상태의 특징
1. 비동기성 : 시간의 순서나 시점에 영향을 받음 / 응답이 오기까지 지연이 존재
2. 오류 처리 : 항상 성공하지 않음. 외부 조작에 의한 정합성 훼손등등...
3. 일관성 : 로컬상태를 서버상태에 동기화하는것
4. 캐싱 : 서버 상태를 로컬에 저장하여 성능 향상을 도모할 수 있지만 데이터 불일치 / 무효화등의 문제가 발생가능
5. Optimistic UI: 사용자가 업데이트시 성공했을때의 데이터를 로컬 상태에서 먼저 표현 / 하지만 실패시 잘 복구해줘야함

서버상태오 로컬상태의 분리 및 비동기성으로 인해 흐름 파악의 어려움 / 예외처리의 어려움이 있다.
이런 어려움 상태관리가 컴포넌트 내용의 대부분이다.

------------------------------------------------------------------------------------

예외흐름 (unhappy path)
- 개발시 계획대로 동작하여 최적의 사용자 경험을 제공하는 정상흐름을 1차적으로 고려
- 하지만 현실은 그렇게 아름답지 않아 예외흐름을 고려하지 않으면 지옥이 된다.

에러 바운더리를 설정하지 않으면 일부 국소 지역에서 발생한 에러가 전체 어플리케이션을 crash로 유도한다.
리액트의 에러바운더리가 내부에서 발생한 js에러를 잡아 로그 기록하고 fallback ui를 표시해줌

사용자의 예측하지 못한 행동도 고려해야함.....
